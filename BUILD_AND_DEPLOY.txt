PICTURE TO SMS — BACKEND + APK

1) DEPLOY THE API (so the app works on any Wi‑Fi / mobile data)

   Option A — Vercel (recommended)
   - Import this GitHub repo in Vercel.
   - Framework preset: Other (no special framework required).
   - Root directory: project root (PictureToSMS), not server.
   - Add environment variables in Vercel project settings:
       OPENROUTER_API_KEY = primary OpenRouter key
       OPENROUTER_API_KEY_2, _3, _4 = optional extra OpenRouter keys (tried in order if the previous fails)
       OPENROUTER_MODEL = single OpenRouter model id (optional; overrides the default chain below)
       OPENROUTER_MODELS = comma-separated list, tried in order (if set, OPENROUTER_MODEL is ignored)
       Example — only flagship: OPENROUTER_MODEL=google/gemini-2.5-pro
       Fallback chain: OPENROUTER_MODELS=google/gemini-2.5-pro,google/gemini-2.5-flash
       Browse ids: https://openrouter.ai/models
       If you omit both, the server defaults to an OpenRouter chain: Google Gemini ids first, then cheaper non-Google vision models (gpt-4o-mini, Llama 3.2 11B vision, gpt-4o, Claude 3 Haiku). Quota/rate/404 skips to the next model in the list.
       Direct Google Gemini (when GEMINI_API_KEY is set) tries a **model chain** on quota/rate/503 (default: 2.5 Flash variants → 2.0/1.5 Flash → 2.5 Pro). Override with GEMINI_MODEL (one id) or GEMINI_MODELS (comma-separated, tried in order).
       After changing env on Vercel: redeploy (or “Redeploy” from the latest deployment) so functions pick up new vars.
       Vercel → Project → Settings → Environment Variables: enable for Production (and Preview if you use preview URLs).
       Note: the server tries Google Gemini (direct API) before OpenRouter when GEMINI_API_KEY is set.
       To actually use OpenRouter for vision, remove or unset GEMINI_API_KEY in that environment (keep only OPENROUTER_API_KEY).
       GEMINI_API_KEY = primary Gemini key (tried first; then OpenRouter, then DeepSeek if set)
       GEMINI_API_KEY_2, _3, _4 = optional extra Gemini keys (same order)
       CLIENT_BEARER_TOKEN = any long random secret (recommended)

   Optional — Google Document AI (server-side “document scanner” quality for burst camera photos)
   The phone stays on fast burst capture; each /v1/analyze call can run Document AI first (deskewed page
   image when the processor returns it), then OpenRouter/Gemini. Not the ML Kit on-phone UI — same GCP family.

   A) In Google Cloud Console (one-time)
   - Pick or create a project. Note the Project ID (not only the display name).
   - APIs & Services → Enable APIs → enable “Cloud Document AI API”.
   - IAM & Admin → Service Accounts → Create account (e.g. picturetosms-docai) → Create key → JSON.
   - Open JSON in an editor; you need client_email and private_key in that file.
   - Document AI → Processors → Create processor → type: Enterprise Document OCR → Region: e.g. United States
     (must match DOCUMENT_AI_LOCATION below, usually us). After creation, open the processor and copy its ID
     (long hex string from the resource name or URL).

   B) Grant the service account access to Document AI
   - IAM → Grant access → add the service account email → role: “Document AI API User” (roles/documentai.apiUser).

   C) Add these environment variables (Vercel / Railway / server .env)

       Local template: edit server/.env (gitignored), then from the server folder run npm run start:local
       so Node loads it (--env-file). Copy the same variable names into Vercel.

       DOCUMENT_AI_PROJECT_ID = same as GCP project id
       DOCUMENT_AI_LOCATION = us   (or eu, etc. — must match where you created the processor)
       DOCUMENT_AI_PROCESSOR_ID = the processor id from the console (hex string)

       Provide credentials in ONE of these ways:
       GOOGLE_SERVICE_ACCOUNT_JSON = entire JSON file as ONE line (minify: remove line breaks inside the file, or use jq: jq -c . service-account.json on Mac/Linux)
       OR
       GOOGLE_SERVICE_ACCOUNT_JSON_BASE64 = base64 of the raw JSON file bytes (easier in Vercel UI on Windows:
         PowerShell: [Convert]::ToBase64String([IO.File]::ReadAllBytes("C:\path\service-account.json")) )

       DOCUMENT_AI_ENABLED = true   (optional; omit to use Document AI whenever the vars above + credentials are set.
         Set to false to disable without deleting secrets.)

       Optional aliases: GOOGLE_CLOUD_PROJECT or GCP_PROJECT can be used instead of DOCUMENT_AI_PROJECT_ID if you prefer.

   D) Redeploy the API (Vercel deploy or Railway redeploy) so the new env vars load.

   Processor types: "Enterprise Document OCR" returns a deskewed raster in pages[].image (best for this app).
   A "Custom Extractor" mainly returns entities/text; if pages[].image is empty, the server skips Document AI for
   that request and uses the burst photo + sharp as before. Fix: add a second processor (Enterprise Document OCR)
   and point DOCUMENT_AI_PROCESSOR_ID at it, or deploy a processor version and set DOCUMENT_AI_PROCESSOR_VERSION.

   If anything is wrong, the server logs [Document AI] and falls back to the normal image + OCR_IMAGE_ENHANCE.

   - Deploy, then copy the public HTTPS URL (e.g. https://pictogpt.vercel.app).
   - The app endpoint is already mapped:
       POST /v1/analyze -> /api/analyze
       GET /health -> /api/health

   Option B — Railway (alternative)
   - Create a new project, add a service from this repo.
   - Set the service root folder to: server
   - Build uses server/Dockerfile automatically when Dockerfile is present.
   - Variables (service -> Variables):
       OPENROUTER_API_KEY (and optional OPENROUTER_API_KEY_2, _3, _4)
       Optional: OPENROUTER_MODEL or OPENROUTER_MODELS (same as Vercel list above)
       GEMINI_API_KEY (and optional GEMINI_API_KEY_2, _3, _4)
       CLIENT_BEARER_TOKEN = any long random secret (recommended)
       Same Document AI variables as in Option A if you use server-side deskew (GOOGLE_SERVICE_ACCOUNT_JSON or
       GOOGLE_SERVICE_ACCOUNT_JSON_BASE64, DOCUMENT_AI_PROJECT_ID, DOCUMENT_AI_LOCATION, DOCUMENT_AI_PROCESSOR_ID)
   - Deploy, then copy the public HTTPS URL (e.g. https://something.up.railway.app).

   Option C — run on your PC for testing only
   - cd server
   - copy .env.example to .env and fill values
   - npm install && npm start
   - Your phone must reach that PC (same LAN + firewall); not ideal outside home.

   Health check: GET https://YOUR_URL/health  →  {"ok":true}

2) BUILD THE APK (Expo EAS — needs a free Expo account)

   From the PictureToSMS folder (this app, not server):

   npm install
   npx eas login
   npx eas init          (links the app once; follow prompts)
   npm run eas:apk       (same as: eas build --platform android --profile preview)

   When the build finishes, open the build page in the browser and download the .apk
   (profile "preview" in eas.json is set to APK, not Play Store bundle).

   CI / no browser: create an EXPO_TOKEN at expo.dev, then:
   set EXPO_TOKEN=...   (PowerShell: $env:EXPO_TOKEN="...")
   npx eas build --platform android --profile preview --non-interactive

3) ON THE PHONE AFTER INSTALL

   - Open the app.
   - Backend API URL = your deployed URL (no trailing slash required).
   - App secret = same value as CLIENT_BEARER_TOKEN (leave blank only if you did not set it on the server).
   - SMS number + prompt as before (SMS is sent automatically on Android by the app).

4) LOCAL APK WITHOUT EAS (if you have Android Studio + JDK)

   - Install JDK 17+, set JAVA_HOME, install Android SDK (ANDROID_HOME).
   - npx expo prebuild --platform android
   - cd android && gradlew.bat assembleRelease
   - APK path: android\app\build\outputs\apk\release\app-release.apk
   - You must sign the release (configure signing in Android docs) for a installable release build;
     EAS handles signing for you when you use eas build.
